1Executive Summary

Bottom line: Component-level tokens are not an over-engineering choice. They are the industry standard for any design system that needs to support multiple themes, dark mode, white-labeling, or multi-platform delivery — all of which HighLevel requires.

Every major design system operated at scale by Google, Microsoft, IBM, Adobe, Salesforce, GitHub, SAP, Alibaba, and GitLab has converged on exactly the same architectural decision: a three-layer token hierarchy that ends in component-specific tokens. The W3C Design Tokens Community Group released their first stable specification in October 2025, and it explicitly supports component-level token structures.

This report surveyed 11 publicly documented design systems that implement component-level tokens. Every system that supports dark mode, multi-brand theming, or third-party customization has found that two layers of tokens are not enough — and has added a third.

HighRise's component token system covers 65 components (38 web, 27 mobile) with full dark/light mode support, 8 size scales, and state coverage across default, hover, active, focused, and disabled interactions. This puts HighRise at or ahead of every industry peer reviewed in this report.

The upfront implementation cost of component tokens is real. So is the long-term cost of not having them: every theme request, every dark mode bug, every white-label inquiry, and every "can we change just this one button?" becomes a manual, error-prone code change instead of a token swap.

Component tokens are not a nice-to-have. They are the foundation of a themeable, maintainable, scalable UI system.

2The Token Hierarchy: A Quick Primer

Design tokens exist in three layers. Each layer references the one above it. The chain is:

Component token → Semantic token → Primitive token → #0042A8
Layer Name Purpose HighRise Example Resolved Value
Layer 1 Primitive token Stores the actual value. Never used directly in components. color.primary.blue.600 #155EEF
Layer 2 Semantic token Names the intent. Used for general UI patterns. Switches between light/dark. color.background.primary.default → {color.primary.blue.600}
Layer 3 Component token Scopes the intent to a specific component, variant, and state. button.color.primary.primary
.background.hover		 → {color.background.primary.intense.1}

Why three layers and not two?

Semantic tokens solve the "what does blue mean?" problem — they replace #155EEF with color.background.primary.default. But semantic tokens are still global. They describe intent across the entire UI. They don't describe what a button's background should be on hover vs. what a tooltip's background should be on hover. Those two things can look the same today and diverge tomorrow. Component tokens give each component its own named contract.

🎨
Primitive tokens

The paint colors in the hardware store. Raw values only.

🗂️
Semantic tokens

The color palette on your wall chart. "Warm beige for living rooms."

🏷️
Component tokens

The label on the actual wall. "Left wall: warm beige, trim: semi-gloss white."

You can repaint any wall without touching the others. That's the point.

🔴 Adobe — Spectrum CSS

Adobe introduces a concept worth adopting: the mod token layer. In addition to internal component tokens, Spectrum CSS exposes a public API of --mod-* tokens that consumers can override without touching internals:

/* Internal component token — owned by the design system */
--spectrum-actionbutton-border-color-default: var(--spectrum-gray-300);

/* Mod token — the public override API */
--mod-actionbutton-border-color: var(--spectrum-actionbutton-border-color-default);

A consumer app only needs to override --mod-actionbutton-border-color. They never touch the internal token. This separates the design system's internal implementation from the customization API — a pattern HighRise could formalize for enterprise white-label use cases.

3.3 The W3C Specification

The W3C Design Tokens Community Group (DTCG) published the first stable version of the Design Token Specification in October 2025. It explicitly supports named token groups (component-scoped namespaces), alias tokens, and the $extends property for component-level token inheritance.

Tools including Figma, Penpot, Sketch, Framer, Supernova, Zeroheight, and Style Dictionary are all implementing this specification. Component-level tokens are not a cutting-edge experiment — they are the established, standardized foundation of modern design systems.

4Why Component Tokens? The Full Case

4.1 Why They Are Required

The fundamental problem semantic tokens cannot solve:

Semantic tokens are global. color.background.primary.default means "the default background for primary UI elements." That definition applies everywhere. But components have unique visual behavior that doesn't always align with a global semantic rule.

Consider hover state — four components, four different behaviors:

  • A button on hover darkens its fill slightly
  • A tooltip on hover may not change at all (it's already visible)
  • An input field on hover shows a darker border, not a different fill
  • A nav item on hover shows a subtle fill change

All four can use the same semantic hover token today. But the moment your design team decides that button hover should be 10% darker than nav hover — a totally valid, common design decision — you have no token to express that. Without component tokens, you either:

  1. Add a new semantic token (which is now component-specific in everything but name), or
  2. Write hard-coded CSS for one component (which breaks your token pipeline)

Both are the same problem: you needed component tokens and didn't have them.

Component tokens are required when:

  • The design system needs to support dark mode cleanly
  • Different products or teams need to customize individual component appearances
  • Components share the same semantic intent but need different visual expressions
  • Mobile and web versions of the same component need different sizing or spacing
  • The system needs to support white-labeling or multi-brand theming

All of these are HighLevel requirements. All of them become token operations — not code changes — when component tokens exist.

4.2 What They Are Used For

Component tokens serve six primary functions:

1. State management

Each interactive state gets its own token. State-specific styling is explicit, testable, and changeable without side effects.

button...background.default  → #0050C8
button...background.hover    → #0042A8
button...background.active   → #003389
button...background.disabled → #C9D8F8
2. Variant isolation

Primary, secondary, tertiary, ghost, error, warning — each variant gets separate tokens. Changing one variant's hover color never affects any other.

3. Size scaling

Every height, padding, gap, and icon size is a named token — not a hard-coded pixel value.

button.regular.height.3xs  → 24px
button.regular.height.sm   → 36px
button.regular.height.xl   → 48px
4. Theme switching

Because component tokens reference semantic tokens, and semantic tokens have light and dark variants, flipping a theme requires changing nothing in the component token layer. The chain resolves automatically.

5. Cross-platform parity

HighRise maintains both web-components/button.json and mobile-components/button.json. Mobile and web have different padding and touch targets while sharing semantic color and typography tokens.

6. White-labeling & multi-brand

Enterprise customers at HighLevel may need custom branded versions. Component tokens make that possible without forking code — only token values change.

4.3 How They Are Implemented

The reference chain

Component tokens work by chaining references through three files:

tokens/web-components/button.json

{
  "button": {
    "color": {
      "primary": {
        "primary": {
          "background": {
            "hover": {
              "value":
  "{color.background
    .primary.intense.1}",
              "type": "color"
            }
          }
        }
      }
    }
  }
}

tokens/Semantics/Light.json

{
  "color": {
    "background": {
      "primary": {
        "intense": {
          "1": {
            "value":
  "{color.primary
    .blue.700}",
            "type": "color"
          }
        }
      }
    }
  }
}

tokens/Primitive.json

{
  "color": {
    "primary": {
      "blue": {
        "700": {
          "value": "#0042A8",
          "type": "color"
        }
      }
    }
  }
}

When dark mode is active, Light.json is swapped for Dark.json — which maps color.background.primary.intense.1 to a different primitive. The component token does not change at all. It just resolves to a different final value.

Tooling

No developer writes component token names manually during day-to-day work. The token pipeline handles this:

ToolRole
Style DictionaryTransforms JSON token files into CSS custom properties, SCSS, iOS Swift, Android XML, or JavaScript constants
Tokens Studio / Figma VariablesSyncs tokens from JSON into Figma's native variable system
HighRise custom Figma pluginApplies component tokens to Figma components via the API
VS Code / IDEAutocompletes token names from the generated CSS output

The output of a component token file is a flat list of CSS custom properties that developers use directly:

:root {
  --button-primary-background-default: #0050C8;
  --button-primary-background-hover:   #0042A8;
  --button-primary-background-active:  #003389;
  --button-primary-background-disabled:#C9D8F8;
  --button-regular-height-sm:   36px;
  --button-regular-padding-x-sm: 12px;
}

.btn-primary {
  background: var(--button-primary-background-default);
  height: var(--button-regular-height-sm);
  padding: 0 var(--button-regular-padding-x-sm);
}
.btn-primary:hover {
  background: var(--button-primary-background-hover);
}

The CSS file itself never changes when themes switch. Only the variable values do.

4.4 Advantages

👩‍💻 For developers
  • One token name per visual property — no guessing which semantic token to use
  • Changing component appearance in any theme is a data change, not a code change
  • Fewer !important overrides and specificity battles
  • Dark mode works automatically — zero per-component dark mode CSS to write
🎨 For designers
  • Every Figma variable has a direct 1:1 counterpart in code
  • Design intent is preserved across handoff — no translation required
  • Theme previews in Figma (light/dark/compact) reflect real implementation behavior
📊 For product & business
  • White-labeling becomes a configuration exercise, not a development project
  • Adding a new theme requires defining token values, not rewriting component CSS
  • Consistent appearance across web, iOS, and Android — enforced by the token layer
🔍 For QA & scale
  • Token values are auditable — "Is button hover color correct in dark mode?" = a single lookup
  • Visual regressions detectable at the token diff level before deployment
  • A system serving 10 products, 3 platforms, 2 themes cannot survive on semantic tokens alone

4.5 Pros and Cons

This is an honest assessment. Component tokens have real costs.

AreaProCon
Control Pixel-precise control over every visual property in every state Large surface area to define and maintain upfront
Theming Theme switching (light/dark/compact/brand) is automatic once tokens are wired Token naming must be stable — renames require migrations
Customization Any team can override any component's appearance without touching shared CSS Inconsistent overrides by downstream teams can fragment the design language
Handoff Figma-to-code mapping is 1:1 and unambiguous Requires design and engineering to agree on token structure before implementation
Dark mode Dark mode is free — no component-level dark mode CSS needed Semantic token layer must be designed correctly for this to work
Tooling Style Dictionary, Tokens Studio handle the heavy lifting Teams must invest in pipeline setup, CI integration, and documentation
Debugging "Which token controls this?" is always answerable Token chains are 3–6 levels deep — tracing a value requires tooling support
Mobile/web Separate files allow platform-appropriate sizing while sharing semantic values Double the component token files to maintain (web + mobile)

Net verdict: The cons are front-loaded (upfront investment) and the pros compound over time. The longer a product runs without component tokens, the more expensive adding them later becomes — because existing components have baked-in assumptions that tokens would have made explicit.

5Addressing the Objection: "They're Too Deep"

The frontend team's concern is understandable and worth addressing directly. Here is the honest response to each objection:

True. button.color.primary.primary.background.hover is 6 levels deep. But consider what it replaces: a hard-coded #0042A8 buried in a component SCSS file, or a semantic token used directly in component CSS — which creates a rigid dependency that cannot be safely overridden.

The long name is the precision. And importantly: developers don't type token names directly in most modern workflows.

  • In Figma, they're variables you click
  • In VS Code, they're autocompleted from generated CSS custom properties
  • In component CSS, the variable is short: var(--button-primary-background-hover)

True for the first component. Not true thereafter — patterns become clear, tooling accelerates the work, and Figma Variables sync handles the design side. The 11 design systems in this report all paid this upfront cost. None of them are reversing course.

A more useful question: how long does it take to not implement component tokens and then need to add dark mode across 65 components later? The answer is: much longer.

It depends on what HighLevel is building. If the answer to any of these questions is yes, then component tokens are necessary:

  • Will HighLevel ever support dark mode? (Yes)
  • Will any component ever need different styling in different contexts? (Yes)
  • Will any enterprise customer ever request custom branding? (Likely)
  • Will the product run on iOS or Android with platform-consistent UI? (Yes — mobile tokens already exist)
  • Will the design system be maintained by multiple teams? (Yes)

As Section 3 demonstrates: they do. Google, Microsoft, IBM, Adobe, Salesforce, GitHub, SAP, Alibaba, GitLab all do. The companies that don't — Atlassian, Shopify, Radix — have either acknowledged the gap (Shopify opened a public GitHub discussion on component tokens, now archived) or operate in a narrower single-product context.

The systems that went deepest — SAP, Salesforce, Google, Microsoft — did so because they serve customers who need customization as a product feature. HighLevel, as a platform serving agencies and businesses, is in exactly this position.

7The Cost of Not Doing Component Tokens

Every shortcut taken now compounds later. Here is what deferring component tokens means in practice:

🌑 Dark mode becomes a CSS marathon

Without component tokens, adding dark mode means writing @media (prefers-color-scheme: dark) or .dark class overrides for every component, every variant, and every state — individually. For 65 components with 5 states and 4 variants each, that is approximately 1,300 individual dark mode CSS rules to write, test, and maintain. With component tokens: zero.

🎭 Theme requests become development projects

"Can we have a compact version of the UI?" Without component tokens: build a separate set of CSS overrides for every spacing property across every component. With component tokens: define a new set of size token values and apply them via a class or data attribute.

🍴 White-labeling requires a code fork

An agency running on HighLevel wants their branded version. Without component tokens: give them a CSS override file they maintain separately (which breaks on every update) or fork the codebase. With component tokens: give them a token values JSON. Every design system update propagates to their brand automatically.

🔀 Cross-team consistency breaks down

When semantic tokens are used directly in component CSS, every team makes their own decisions about which semantic token to use for which component state. Over time, the same button looks different across Team 1's product and Team 2's product. Component tokens are the single source of truth that prevents this.

📈 Technical debt compounds

The longer a product runs without component tokens, the harder they become to add. Every component that has baked-in semantic token references or hard-coded values becomes a migration project. Starting with component tokens means never having that conversation.

8Glossary

TermDefinition
Design tokenA named variable that stores a visual design decision (color, size, spacing, typography). The single source of truth between design and code.
Primitive tokenThe lowest layer. Stores raw values: color.primary.blue.600 = #155EEF. Never referenced directly in components.
Semantic tokenThe middle layer. Names the intent: color.background.primary.default. References a primitive. Changes value when theme changes.
Component tokenThe top layer. Scopes the intent to a specific component and state: button.color.primary.primary.background.hover. References a semantic token.
AliasA token that references another token instead of a raw value. All semantic and component tokens are aliases.
CSS custom propertyA browser-native variable, written as --variable-name: value. The most common output format for design tokens on the web.
Style DictionaryAn open-source tool by Amazon that transforms design token JSON files into platform-specific outputs (CSS, SCSS, iOS Swift, Android XML, JavaScript constants).
Tokens StudioA Figma plugin that syncs design token JSON files with Figma's native variable and style system.
Token pipelineThe automated process that takes token source files (JSON), transforms them via Style Dictionary, and outputs platform-specific variables for developers to use.
Dark modeA UI theme where backgrounds are dark and text is light. With component tokens, dark mode requires no component-level CSS changes — only semantic token values change.
White-labelingProviding the same product with a different brand identity to a different customer. Enabled by changing token values without changing code.
DTCGDesign Tokens Community Group — the W3C working group that published the first stable design token specification in October 2025.
